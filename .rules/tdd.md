# TDD ルール

## 開発サイクル

実装前にテストを書く（Red → Green → Refactor）。

1. **Red**: 失敗するテストを書く
2. **Green**: テストが通る最小限の実装を書く
3. **Refactor**: テストを壊さずにコードを整理する

## テストの配置

```
tests/
  unit/        ← 純粋な関数・クラス（IO なし）
  integration/ ← CSV 読み込み・ファイル書き込みを含むもの
  fixtures/    ← テスト用 CSV（実計測データを流用してよい）
```

テストファイル名は `test_<モジュール名>.py`。

## フィクスチャ

`data/raw/` の実計測 CSV を `tests/fixtures/` にコピーして使ってよい。
ただし fixture は最小限（数十行程度）に絞ること。大量データを commit しない。

実データには 2 つのフォーマットが混在する:
- 旧: `Timestamp`, `Temperature (℃)`, `Humidity (%)`
- 新: `timestamp`, `temperature_c`, `humidity_pct`

ロジックがフォーマット差を吸収するなら、両フォーマットのケースをテストに含める。

## テスト対象の優先順位

1. **必ずテストする**: データ変換・集計・リサンプリング・ドメイン計算など純粋なロジック
2. **できる範囲でテストする**: CSV 読み込み・ファイル書き込み（integration テスト）
3. **テストしない**: Streamlit の描画コード、センサー sysfs 読み取り（ハードウェア依存）

Streamlit のページスクリプトにロジックを直書きしない。テスト可能な関数に切り出してから呼ぶ。

## テストランナー

```bash
pytest tests/
```

CI がない場合も、実装前に手元で Red を確認してから Green にすること。

## やらないこと

- テストなしで実装を先行させない
- 実装に合わせてテストをあとから書く（テストが実装の追認になる）
- 実データファイルをそのまま大量に commit する
